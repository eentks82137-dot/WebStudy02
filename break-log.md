# 에러 만들어보기

## **실험 1: Accept 헤더 변조**

- 행동: 부메랑(또는 JavaScript fetch)으로 `/hw03/convert?from=km&to=mile&value=10` 요청을 보내되, `Accept: application/xml`로 설정한다.
- 관찰: 어떤 상태 코드와 메시지가 오는가?
- 핵심 질문: 서버가 지원하지 않는 형식을 요청하면 어떤 상태 코드가 적절한가? 이것이 Content Negotiation의 본질이다.

---

- 변경 내용: `Accept` 헤더를 `application/xml`로 설정하여 서버가 지원하지 않는 형식을 요청한다.
- 예상 결과: 406 Not Acceptable 코드와 함께 지원하지 않는 Accept 헤더입니다라는 메세지가 반환될 것이다.
- 관찰 결과: `406 <h1>Error</h1><p>지원하지 않는 Accept 헤더입니다</p>` 라는 응답이 왔다.
- 원인 분석: 서버에서 Accept 헤더를 검사할 때 html과 json만 지원하도록 되어 있기 때문에
- 검증 방법: js로 fetch를 이용하여 `Accept: application/xml`로 요청을 보내고 응답을 확인한다.

## **실험 2: Accept-Language 변조**

- 행동: 브라우저 언어 설정을 영어(en-US)로 바꾸고 접속한다. 그다음 부메랑으로 `Accept-Language: ja-JP`를 보내본다.
- 관찰: UI 라벨과 숫자 포맷이 바뀌는가? 지원하지 않는 언어(ja-JP)일 때는 어떤 언어로 표시되는가?
- 핵심 질문: 지원하지 않는 언어가 요청되면 서버는 어떻게 대응해야 하는가? fallback 전략이란 무엇인가?

---

- 변경 내용: 부메랑에서 Accept-Language 헤더를 ja-JP로 설정하여 지원하지 않는 언어를 요청한다.
- 예상 결과: UI 라벨과 숫자 포맷이 영어로 표시될 것이다.
- 관찰 결과: UI 라벨과 숫자 포맷이 영어로 표시되었다.
- 원인 분석: 서버에서 Accept-Language 헤더를 검사할 때 ja-JP는 지원하지 않는 언어이므로 기본 언어인 영어로 반환이 된다.
- 검증 방법: 부메랑에서 `Accept-Language: ja-JP`로 요청을 보내고 UI 라벨과 숫자 포맷이 영어로 표시되는지 확인한다.

## **실험 3: 잘못된 입력값**

- 행동: `/hw03/convert?from=km&to=mile&value=abc`로 요청을 보낸다. 한 번은 `Accept: text/html`로, 한 번은 `Accept: application/json`으로 보낸다.
- 관찰: 두 경우 모두 400 응답이 오는가? HTML 요청에는 HTML 에러 페이지가, JSON 요청에는 JSON 에러 메시지가 오는가?
- 핵심 질문: 에러 응답에도 Content Negotiation이 적용되어야 하는 이유는 무엇인가? 만약 JSON을 기대하는 클라이언트에게 HTML 에러 페이지가 오면 어떤 문제가 생기는가?

---

- 변경 내용: value 파라미터를 숫자가 아닌 문자로 설정하여 요청을 보낸다.
- 예상 결과: value는 숫자여야 한다는 메세지가 출력된다
- 관찰 결과: value는 숫자여야 한다는 메세지가 출력되었다. Json으로 요청했을 때는 아래와 같이 에러 메세지가 반환되었다.

```json
{
  "error": "Bad Request",
  "status": 400,
  "message": "value는 숫자여야 합니다: abcc"
}
```

- 원인 분석: 서버에서 value 파라미터를 숫자로 변환할 때 예외가 발생하여 해당 응답이 반환된다.
- 검증 방법: `/hw03/convert?value=abcc&from=mile&to=km`로 요청을 보내고 응답을 확인한다.

## **실험 4: 단위 조합 오류**

- 행동: `/hw03/convert?from=km&to=C&value=100`으로 요청을 보낸다.
- 관찰: 어떤 상태 코드가 오는가? 에러 메시지는 무엇인가?
- 핵심 질문: 이 에러는 406(Not Acceptable)인가, 400(Bad Request)인가? 406은 "응답 형식을 제공할 수 없다"이고, 400은 "요청 데이터가 잘못되었다"이다. 이 두 상태 코드의 의미 차이를 명확히 하라.

---

- 변경 내용: from과 to 단위를 서로 호환되지 않는 조합으로 설정하여 요청을 보낸다
- 예상 결과: 400 Bad Request 코드와 함께 지원하지 않는 단위 변환이라는 메세지가 반환될 것이다.
- 관찰 결과: 400 Bad Request 코드와 함께 지원하지 않는 단위 변환이라는 메세지가 반환되었다.
- 원인 분석: 406 Not Acceptable은 클라이언트가 요청한 형식으로 응답을 제공할 수 없을때 사용되는 코드이고 400 Bad Request는 클라이언트의 요청이 잘못되었을 때 사용되는 코드이다. 이와 같은 경우에는 클라이언트의 요청이 잘못되었으므로 400 Bad Request가 적절하다.
- 검증 방법: `/hw03/convert?from=km&to=c&value=100`로 요청을 보내고 응답을 확인한다.

## **실험 5: Accept-Language 헤더 제거**

- 행동: 부메랑으로 Accept-Language 헤더 없이 요청을 보낸다.
- 관찰: UI 라벨은 어떤 언어로 표시되는가? 숫자 포맷은 어떤 로케일 기준인가?
- 핵심 질문: Accept-Language 헤더가 없으면 `request.getLocale()`은 무엇을 반환하는가? 서버의 기본 로케일인가? 이것이 Day 6의 질문("사용자의 기본 로케일은 서버 기준인가 클라이언트 기준인가?")과 어떻게 연결되는가?

---

- 변경 내용: 부메랑에서 Accept-Language 헤더를 제거하여 요청을 보낸다.
- 예상 결과: UI 라벨은 영어로 표시되고 숫자 포맷도 영어로 표시될 것이다.
- 관찰 결과: Accept-Language 헤더의 값을 제거하여 요청을 보내보았을 때는 en_US로 표시되었고, Accept-Language 헤더 설정 자체를 제거하여 요청을 보내보았을 때는 ko로 표시되었다.
- 원인 분석: Accept-Language 헤더가 없으면 서버는 기본 로케일을 사용하여 응답을 생성한다. wsl에서 기본 로케일은 영어로 설정되어 있기 때문에 영어로 표시된다.
- 검증 방법: 부메랑에서 Accept-Language 헤더의 값을 제거하여 요청을 보내보고, Accept-Language 헤더 설정 자체를 제거하여 요청을 보내보고 UI 라벨과 숫자 포맷이 영어로 표시되는지 확인한다.
